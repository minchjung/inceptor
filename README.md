### 태스크 카드 설정 Tip 
- 기능을 **세분화** 예상 소요 시간을 3시간 이하  
- 기능별 세분화 테스크 부재시  프로젝트 방향성을 잃어버리기 쉽다
- 일주일에 코딩 시간은 25시간 정도 (25 x 4명 = 100)  
- 능력에 맞춰서 개발 시간을 측정해보는 것도 프로젝트의 목적 중 하나
- 시간이 어긋나는 부분에 대해서 너무 부담가지지 않으시길 바랍니다

# 5. Not To Do 리스트 점검 
- 하지 말아야 할 일만 제대로 지켜도 프로젝트는 실패하지 않음
- SR 피드백이 진행될 때에, 이 사전 체크리스트를 충실하게 따랐는지 확인 
- 다음 항목을 차근차근 검토해보세요 

> 어느 날 짐 콜린스가 이런저런 고민을 털어놓자 피터 드러커가 조용히 물었다.
> "해야 할 일(To do)리스트가 있습니까?"
> "예, 그럼요"

> 콜린스는 당연하다는 듯 대답했다. 그러자 드러커가 기다렸다는 듯 다시 물었다.
> "그러면 하지 말아야 할 일(Not to do) 리스트도 있겠지요?"

> <시작하라 그들처럼> 서광원
>해야 할 목록을 만드는 것이 아니라, 하지 않아도 될 일 목록(Not to do list)을 만드는 것이 중요합니다. 우리는 해야 할 일 목록(To do list) 작성은 잘하는데, 하지 않아도 될 일의 목록을 만드는 데는 서툽니다. 진정으로 변화를 원한다면 먼저 불필요한 일을 해야 할 일의 목록에서 덜어내야 합니다.

> <초격차> 권오현
## 사전 체크 리스트 공통 
### 코딩 앞서 
ㅤ⬜  기획이 마무리되지 않은 상태에서 코딩을 시작함  
### 아키텍처
ㅤ⬜ 논의되지 않은 새로운 스택 사용    
ㅤ⬜ 핵심 기능이 외부 API에만 의존함 (api 사용량을 고려하지 않음)  
ㅤ⬜ 인증/권한 사용시 passport.js를 사용함 (여러가지 이유로 권장하지 않습니다)   
### 사전 체크리스트 백앤드 
 - **API**  
ㅤ⬜ 리소스 CRUD시 목적에 맞는 HTTP 메소드를 사용하지 않음  
ㅤ⬜ 리소스 CRUD시 엔드포인트에 동사를 사용함   
ㅤ⬜ 리소스 요청/응답시 필요한 json의 키/값 타입을 명시하지 않음  
ㅤ⬜ 리소스 여러 개 조회 시 배열을 사용하지 않음  
ㅤ⬜ HTTP 응답 코드의 사용이 적절하지 않음  
ㅤ⬜ 응답 에러 메시지의 일관성이 없음    
ㅤ⬜ 리소스 조회 시 json과 스키마 field간의 일관성이 없음  
ㅤ⬜ 리소스 생성 시 신규 아이디를 리턴하지 않음    
ㅤ⬜ 사용자 정보를 필요로 하는 요청에 토큰 또는 세션을 사용하지 않음  
ㅤ⬜ 특정 사용자와 연관된 리소스 조회시 권한 문제를 고려하지 않음    
ㅤ⬜ JWT 사용시 헤더에 토큰을 전달하는 내용을 명시하지 않음  
ㅤ⬜ 엔드포인트가 spinal-case (소문자와 대시 문자열) 를 따르지 않음  
ㅤ⬜ GET 요청에서 쿼리 파라미터 대신 HTTP body를 사용함  
ㅤ⬜ HTTP body를 사용해야 하는 상황에서 쿼리 파라미터를 사용함  
- **스키마**  
ㅤ⬜ 스키마의 1:N, N:M 관계가 적절하지 않음  
ㅤ⬜ 조인 테이블의 이름이, 관계있는 테이블 이름을 포함하지 않음  
ㅤ⬜ 리소스의 CRUD에 필요한 테이블이 존재하지 않음  
### 사전 체크리스트 프론트엔드 
ㅤ⬜ 이미 전부 디자인 가이드가 만들어져있는 CSS 프레임워크를 사용함 (ex. Material UI, Ant design, bootstrap 등)  
### 사후 채크 리스트 프론트엔드
ㅤ⬜ UX상 나쁜 디자인으로 평가받는 브라우저 내장 기능 alert, prompt, confirm의 지나친 사용  
ㅤ⬜ 콘텐츠가 존재하나 스크롤할 수 없는 영역이 존재함  
ㅤ⬜ "스크롤바 항상보기" 옵션을 사용시, 불필요하게 스크롤바가 보임  
ㅤ⬜ 데이터가 0개일 때, 아무런 안내가 없어 버그처럼 보임  
ㅤ⬜ 로딩 실패 시 적절한 피드백을 제공하지 않음  
ㅤ⬜ 로딩 시 레이아웃이 깨짐  
ㅤ⬜ 로딩 처리를 전혀 하지 않음  
ㅤ⬜ 개발자 콘솔을 열었을 때 자바스크립트 에러가 발생함  
ㅤ⬜ 로그인을 하지 않을 경우 아무런 기능도 사용할 수 없음
